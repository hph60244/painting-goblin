## 第一步：自我盤點與限制設定

> **目標：** 客觀評估自己的資源與限制，作為後續所有決策的邊界條件。

### A. 列出你的限制（時間／技能／資源）

📺 **Tips for Making a Game You'll ACTUALLY Finish** 提供了一個完整框架：**列出限制 → 評估優勢 → 尋找市場缺口**。

| 面向 | 提問 | 具體紀錄 |
|------|------|---------|
| **時間** | 每天／每週有多少開發時間？預計總工時？ | |
| **技能** | 擅長什麼？不擅長什麼？（程式／美術／音樂／設計） | |
| **團隊** | 獨自開發？有合夥人？可以外包什麼？ | |
| **資金** | 有預算買素材／工具嗎？還是全免費？ | |
| **目標** | 做興趣？上 Steam 賣錢？參加 Game Jam？ | |
| **耐力** | 你能對一個點子保持熱情多久？3 週還是 3 年？ | |

**核心洞察：誠實面對你的限制，而不是用理想來欺騙自己。** 如果只有週末時間，就不要計畫做一款 100 小時的 JRPG。

### B. 用「限制」創造創意的邊界

📺 **Design by Constraint** 的核心主張：限制不是障礙，而是創意的催化劑。因為限制**減少決策癱瘓**，讓你不用在無限可能性中迷路。

📺 **Subtractive Design** 進一步提出三種減法策略：減少角色、減少關卡、減少機制。

📺 **The 100 Games That Taught Me Game Design** 告訴我們：每一款遊戲只需要聚焦一個核心設計概念，不需要一次做完所有事。

**設定 3~5 條硬限制，例如：**

> ❌ 不能做：開放世界、多人連線、即時戰鬥、自訂角色、語音對話
> ✅ 只能做：2D 像素、回合制、單螢幕關卡、無文字故事

📺 **Ma, The Space Between** 引入日本「間」的概念：空白和限制本身可以創造更深刻的體驗。去除不必要的元素，留下來的東西會更有力量。

📺 **Procedural Generation** 也提醒：在嘗試自動生成之前，**先手動設計關卡建立規則**，這樣你才知道你要自動化什麼。

### C. 決定你的「無法妥協的設計承諾」

對自己宣告 3 條「無論如何都不會刪除」的設計承諾，這會成為所有後續決策的 anchor。例如：

1. 遊戲必須在 15 分鐘內讓玩家體驗到核心樂趣
2. 遊戲不能有文字（全球化的福音）
3. 遊戲的失敗必須有趣，而不是懲罰

📺 **Designing Heartbreak, Propaganda and Death** 告訴我們：情感體驗可以透過數學公式和條件判斷來設計。如果你想要玩家感受某種情感，**先定義那條情感曲線的數學行為**。

## 第二步：靈感生成與市場探索

> **目標：** 在不脫離第一步限制的前提下，產生多個遊戲點子的候選清單。

### A. 靈感生成方法（選 2~3 種嘗試）

📺 **The Secret to GOOD Game Ideas** 提供了八種方法，📺 **How to find amazing game ideas** 提供了四種來源。以下整合並深化：

#### 方法一：從你自己的生活經驗取材

> 📺 「如果你自己沒做過，就不要寫這個故事。」—— 好的遊戲點子來自真實的個人體驗。

- 你最近學會了什麼？（烹飪、園藝、寫程式）
- 你在生活中遇到了什麼問題？（空間利用、時間管理、社交焦慮）
- 你的職業中有什麼有趣的流程？

**案例：** *Papers, Please* 來自開發者對邊境檢查這個「無聊」工作的想像。*Stardew Valley* 來自開發者對城市生活的厭倦。

#### 方法二：從非遊戲媒介獲取靈感

📺 **The Science Behind Abstract Storytelling** 指出人類大腦有 Apophenia（自動為抽象事物賦予意義的天性）。善用這個特性來設計開放式敘事。

- **文學：** 某個小說的設定或角色關係
- **電影：** 某個場景的視覺風格或情緒
- **桌遊：** 某個機制（資源管理、區域控制）
- **現實生活：** 某次排隊、某個夢、某種交通方式

📺 **What Makes a Game Feel Mysterious** 指出神秘感的來源：鎖住的門、隱藏的規則、未知的地圖、敘事謎團。你可以從其他媒介中辨識這些元素。

#### 方法三：改造現有類型

📺 **How To Combine Video Game Genres** 提出了三種混合方式：

1. **切換法（Switching）：** 不同時間切換不同類型（如 *Yakuza* 平時是 RPG，進戰鬥變格鬥）
2. **遊玩風格法（Play Style）：** 玩家自己選擇想玩哪種風格
3. **融合法（Fusion）：** 將不同類型的元素融合成新玩法

📺 **Throwing Down on Genre** 進一步提醒：**類型分類最終是任意的**。不要被現有的類型標籤限制你的創意。你的遊戲不需要被某個類別框住。

#### 方法四：SCAMPER 技術（針對一個既有遊戲改造）

- **S**ubstitute（替代）：回合制→即時制、像素→3D、單人→合作
- **C**ombine（結合）：RPG + 卡牌、三消 + 城市建造、節奏 + 射擊
- **A**dapt（改編）：把桌遊機制改成電子遊戲、把電影情節改成遊戲世界
- **M**odify（修改）：改變勝利條件、改變視角、改變資源系統
- **P**ut to other use（另作他用）：用戰鬥系統做辯論遊戲、用 RPG 做戀愛模擬
- **E**liminate（刪除）：沒有戰鬥的 RPG、沒有對話的冒險遊戲、沒有敵人只有解謎
- **R**earrange（重組）：讓怪物收集英雄、讓玩家扮演反派

#### 方法五：從體驗出發

📺 **What Sells on Steam You Don't Need a Hook** 的核心論證：**玩家購買的不是功能，而是體驗**。

先問：「我想讓玩家**感受到什麼**？」再反推機制。

| 我想讓玩家感受到… | 可能的遊戲方向 |
|-----------------|--------------|
| 孤獨感 | 太空探索、深海潛水、荒島生存 |
| 掌控感 | 城市建造、帝國管理、公司經營 |
| 探索的驚喜 | Metroidvania、開放世界 |
| 社交溫暖 | Cozy Game、動物森友會類型 |
| 緊張的壓力 | 恐怖遊戲、限時生存 |
| 創造的成就感 | 沙盒建造、料理模擬 |

#### 方法六：從一個「按鈕」開始

📺 **Can We Make This Button Fun To Press** 提出一個原子級的分析方法：

> 從遊戲中最基本的操作單元開始，問：這個操作本身有趣嗎？

從一個操作出發建構樂趣：
1. **衝擊感（Impact）：** 操作的回饋（震動、音效、動畫）
2. **獎勵（Reward）：** 操作後的結果（分數、視覺變化）
3. **挑戰（Challenge）：** 操作的難度（時機、精準度）
4. **奇幻感（Fantasy）：** 操作帶入的情境（你在開槍／施法／跳躍）

**練習：** 寫下你的遊戲中最基本的三個操作，逐一分析它們是否滿足以上四點。

📺 **Nintendo - Putting Play First** 強化了這個觀點：任天堂的哲學就是**先找到一個有趣的互動方式，再圍繞它設計角色、世界、故事**。*Mario* 的跳躍、*Zelda* 的揮劍、*Splatoon* 的射墨——都是從一個操作出發。

### B. 找出市場缺口

📺 **Twitch Kills Game Sales.. but not equally!** 的數據：約 40% 因觀看直播而購買，約 39% 因觀看而放棄購買。你的遊戲需要「直播友好」——有視覺上的高潮點、可討論的決策、或讓觀眾驚喜的時刻。

在 Steam 上搜尋你的方向，問：

1. **這個類型有多少競爭者？**（太多→紅海，太少→可能有市場但也要確認需求）
2. **現有遊戲的評價「抱怨」什麼？**（那就是你的機會）
   - 「美術好看但不好玩」→ 你的機會在玩法
   - 「系統太複雜沒教學」→ 你的機會在易用性
   - 「內容太少」→ 你的機會在規模（但也要注意 scope）
3. **有什麼玩家渴望但沒被滿足的體驗？**
   - 看看 Steam 的「推薦」和「願望清單」評論
   - 看看 Reddit/ Discord 對某個類型的討論

📺 **Tips for Making a Game You'll ACTUALLY Finish** 提醒：**你的優勢領域**在哪裡，就去哪裡找市場缺口。如果你是美術強項，做一個視覺驅動的遊戲；如果你是程式強項，做一個機制驅動的遊戲。

#### 核心洞察：不要追求「獨特的鉤子」

📺 **What Sells on Steam You Don't Need a Hook** 的結論：成功的遊戲通常沒有獨一無二的「鉤子」（Hook），而是**把一個常見的體驗做到極致**。*Vampire Survivors* 不是第一個 Survivor-like，它是第一個做到極致的。你的遊戲不需要是「第一個」，只需要是「最好的之一」。

## 第三步：篩選與聚焦

> **目標：** 從眾多點子中選出一個最適合你的，並削減到最小可行版本。

### A. 篩選標準（加權評分）

📺 **Game Design Theory A Guided Tour** 提供了三個理論視角來評估你的想法：
1. **工藝（Craft）：** 你的技術能力能做嗎？
2. **玩家（Players）：** 有人會想玩嗎？
3. **遊戲本身（Game）：** 這個系統有趣嗎？

| 標準 | 權重 | 評分 (1~5) | 備註 |
|------|:---:|:----------:|------|
| **熱情**：能做半年以上而不厭倦嗎？ | 高 | | 熱情是獨立開發唯一的續航力 |
| **技能匹配**：你的強項能發揮嗎？ | 中高 | | 不要做你不擅長的領域 |
| **限制符合**：符合第一步的限制嗎？ | 高 | | 不符合的直接淘汰 |
| **市場差異**：和既有作品的區別足夠有意義嗎？ | 中 | | 區別不夠 → 玩家選別人就好 |
| **規模可控**：能在時間預算內完成 MVP 嗎？ | 高 | | 做不出來就沒有意義 |
| **X-Factor**：這個點子讓你興奮得睡不著嗎？ | 中 | | 直覺有時候比理性重要 |
| **總分** | | /30 | 低於 20 分建議放棄 |

📺 **Playing Like a Designer** 提醒：在評估時要**意識到自己的偏誤**。你可能對某個點子有情感依戀（創作者偏誤），或者過於樂觀地評估自己的時間和能力（計畫謬誤）。找一個朋友幫你客觀評估。

### B. 削減到 MVP（最小可行產品）

選定點子後，回答：

> **如果我只做 1 個月，能做出什麼可玩的東西？**

📺 **Making Your First Game Minimum Viable Product** 提供了三個經典案例：
- *Super Mario Bros* 的 MVP：只有跳躍和敵人碰撞
- *JRPG* 的 MVP：只有戰鬥循環（遇敵→戰鬥→獎勵）
- *Ikaruga* 的 MVP：只有極性切換和射擊

📺 **Prototyping, Vertical Slices, and Planning a Complex Game** 的功能分類法：
將所有功能分為兩類：

| 類別 | 定義 | 處理方式 |
|------|------|---------|
| **類型標準（Genre Standard）** | 這個類型的玩家期待的功能 | 只做最基本的，夠用就好 |
| **創新功能（Innovation）** | 你的遊戲獨特的地方 | 優先確保它能運作且有趣 |

**練習：** 拿一張紙，左邊寫「類型標準」，右邊寫「創新功能」。把右邊削到只剩 1~2 項。

📺 **Subtractive Design** 的三種減法策略：
- **減少角色：** 不需要多角色就做一個
- **減少關卡：** 不需要 100 關就做 3 關（但要是精華關卡）
- **減少機制：** 不需要 10 種能力就做 2 種（但要是深度夠的 2 種）

📺 **Game Design Documents - a Minimalist Approach** 建議：GDD 應該「先廣後深、面面俱到但保持精簡」。花一天寫一份 3~5 頁的 GDD，涵蓋所有面向（機制、美術、故事、技術），但每一頁都控制在 300 字內。目的不是寫完文件，而是**快速進入原型階段**。

### C. 正面面對「刪除」的心理門檻

📺 **Kill Your Darlings** 提出了一個關鍵心態：**「強烈觀點、軟弱持有」（Strong Views, Weakly Held）**。

- **強烈觀點：** 對你的設計有信心，全力投入
- **軟弱持有：** 準備好隨時放棄它

📺 **Fail Faster** 進一步解釋：**越晚發現問題，修正成本越高**。在原型階段砍掉一個功能只浪費 1 天，在開發末期砍掉浪費 1 個月。

## 第四步：核心機制設計

> **目標：** 確定遊戲的核心玩法循環（Core Gameplay Loop），讓它**本質上有趣**而不是靠獎勵驅動。

### A. 定義核心玩法循環

📺 **Mechanics Shaping Story** 的核心洞察：**你的核心玩法循環不只是「玩法」，它在潛意識中傳達了一套價值觀**。*Pokemon* 的捕捉→戰鬥循環暗示了收集與馴化。*D&D* 的職業系統暗示了個人英雄主義。你的循環在說什麼故事？

回答三個問題：

```
1. 玩家「重複做什麼」？
2. 這個動作「產生什麼回饋」？
3. 這個回饋如何「改變遊戲狀態」並引入新的決策？
```

📺 **Game Mechanics & Systems Thinking** 的定義：
- **機制（Mechanic）：** 原子級的規則（按下跳躍→角色上升）
- **系統（System）：** 多個機制互動的整體（跳躍＋重力＋平台＋敵人＝平台跳躍系統）
- **表達性機制（Expressive Mechanic）：** 機制本身傳達意義（*Papers, Please* 的印章機制傳達官僚壓迫）

**案例：俄羅斯方塊**
```
移動/旋轉方塊 → 方塊落下填滿 → 消除一行得分 → 方塊掉落速度變快（新的挑戰）
```

更好的案例 — 📺 **How to Make Anything Fun** 的 Flow 框架：
- **明確的目標：** 消除行、不讓方塊堆到頂
- **即時回饋：** 每消除一行立即得分和畫面變化
- **挑戰與技巧的平衡：** 速度逐漸加快，對應玩家逐漸熟練

### B. 確保你的「循環」真的有趣

📺 **How do Game Designers make their games FUN** 的 MDA 框架：

| 樂趣類型（Aesthetics） | 你的遊戲如何提供？ | 驗證問題 |
|----------------------|------------------|---------|
| **感官（Sensation）** | 畫面、音效、震動回饋 | 「閉上眼睛玩還是做同樣的事嗎？」 |
| **社交（Social）** | 排行榜、合作、競爭、分享 | 「有人一起玩會更好玩嗎？」 |
| **挑戰（Challenge）** | 需要技巧克服的障礙 | 「失敗時是想再試一次還是放棄？」 |
| **幻想（Fantasy）** | 讓玩家扮演某個角色/情境 | 「你有想像自己是遊戲中的誰嗎？」 |
| **敘事（Narrative）** | 故事推動 | 「你會想知道接下來發生什麼嗎？」 |
| **發現（Discovery）** | 隱藏內容、意外驚喜 | 「你會在遊戲中發現意想不到的東西嗎？」 |
| **表達（Expression）** | 自訂角色、創造內容 | 「你能在遊戲中留下自己的印記嗎？」 |
| **沉浸（Submission）** | 放空、重複性獎勵 | 「你會不自覺地玩了一小時嗎？」 |

📺 **The Skinner Box** 警告：不要用隨機獎勵（Skinner Box）來掩蓋無趣的核心循環。玩家遲早會覺醒。更好的替代方案：
- **好奇心：** 讓玩家想知道「如果我這樣做會怎樣？」
- **精通：** 讓玩家感受到技巧進步
- **挑戰：** 讓玩家想克服障礙
- **故事：** 讓玩家想知道接下來發生什麼

📺 **What is Fun** 提供了「14 種樂趣形式」作為 Design Pillars。選 3~4 個作為你的設計支柱，每個後續決策都要問：「這個決策支援我的設計支柱嗎？」

### C. 用「鏈」代替「循環」思考

📺 **Gameplay Loops Are Out, Chains Are In** 提出了一個深刻的觀點：

> 循環（Loop）暗示了重複和停滯，但鏈（Chain）描述了**情感弧線**和**價值累積**。

不是「重複做 A → 得到 B → 重複做 A」，而是：
```
不懂 → 理解 → 嘗試 → 失敗 → 學到新技巧 → 成功 → 精通 →
發現隱藏內容 → 更深入的理解 → 更高層次的精通
```

**設計「隱藏秘密鏈」：** 讓玩家在後期才發現某些機制／內容的存在，給他們「原來我一直擁有這個能力！」的驚喜感。

📺 **Exploration in Games** 將探索樂趣分為四種類型。每個類別都可以設計一條鏈：

| 探索類型 | 鏈的設計方式 |
|---------|------------|
| **地理發現** | 新區域 → 新挑戰 → 新獎勵 → 更深的區域 |
| **機制發現** | 發現隱藏機制 → 利用它 → 解鎖新策略 → 發現更多機制 |
| **內容發現** | 找到隱藏故事 → 串聯線索 → 理解世界觀 → 更大的謎團 |
| **敘事發現** | 遇到角色 → 了解背景 → 做出選擇 → 看到後果 → 更大轉折 |

### D. 從一個原子操作開始設計

📺 **Can We Make This Button Fun To Press** 的方法論可以進階應用：

**步驟 1：** 定義遊戲中的最小操作單元（跳躍、射擊、點擊、滑動）
**步驟 2：** 為這個操作設計四層體驗：
1. 衝擊感：即時的物理回饋（動畫、音效、震動）
2. 獎勵：操作後的結果（視覺變化、數值變化、狀態變化）
3. 挑戰：操作的難度曲線（時機、精準度、組合）
4. 奇幻感：這個操作讓你「成為」什麼？（英雄、工匠、探險家）

**步驟 3：** 測試這個操作本身是否有趣。如果一個按鈕不好按，整個遊戲都不會好玩。

### E. 選擇你的隨機性類型

📺 **The Two Types of Random in Game Design** 的關鍵區分：

| 隨機性類型 | 定義 | 例子 | 設計意圖 |
|-----------|------|------|---------|
| **輸入隨機性（Input Randomness）** | 先給隨機資訊，玩家再做決策 | *Hearthstone* 的手牌、*Slay the Spire* 的卡牌 | 鼓勵策略思考，玩家感覺「是我贏的」 |
| **輸出隨機性（Output Randomness）** | 玩家先決策，結果由隨機決定 | 命中率 80% 的攻擊 miss | 增加不確定性和戲劇性，但可能感覺不公平 |

**建議：** 優先使用「輸入隨機性」。它讓玩家感覺**技術和策略決定結果**，而不是運氣。

📺 **How Games Twist Probability** 提醒：人類對概率的感知是偏差的。你可以利用這個偏差，也可以用隱藏概率讓玩家的直覺「感覺對」。

## 第五步：玩家體驗設計

> **目標：** 確保你的遊戲不僅好玩，還提供了完整且一致的玩家體驗。

### A. 定義你的目標玩家

📺 **Balancing an MMO Ecosystem** 基於 Bartle 的玩家分類，但更重要的是：**你的遊戲不需要服務所有類型的玩家。**

| 玩家類型 | 追求什麼？ | 你的遊戲能滿足嗎？ | 如果否，接受它 |
|---------|-----------|----------------|---------------|
| **成就者（Achievers）** | 高分、收集、100% 完成度 | | 不需要成就系統 |
| **探索者（Explorers）** | 發現秘密、理解系統 | | 不需要隱藏內容 |
| **社交者（Socializers）** | 與人互動、分享體驗 | | 不需要多人模式 |
| **殺手（Killers）** | 擊敗對手、證明實力 | | 不需要 PvP |

📺 **No Gendered Mechanics** 的批判：別假設某些機制是「給男生」或「給女生」的。是你的**主題**決定了玩家群體，而不是機制。三消可以是硬核策略遊戲，FPS 可以是溫馨社交遊戲。

> **核心洞察：清楚地知道你的遊戲「不是」給誰的。** 試圖討好所有人 = 沒有人滿意。

📺 **The J.C. Penney's Effect** 的教訓：**「感覺公平」比「真正的公平」更重要。** 玩家對感受的評價往往超過對事實的評價。

- 你的獎勵系統可能數學上完全公平，但玩家覺得「為什麼他拿到好東西我沒有？」
- 解決方案：**透明化你的獎勵邏輯**，或者**讓玩家感覺自己「贏」了獎勵**而不是「隨機」得到

### B. 設計難度曲線（挑戰 vs 懲罰）

📺 **When Difficult Is Fun** 提供了關鍵區分：

| 維度 | ✅ 有挑戰性的遊戲 | ❌ 懲罰性的遊戲 |
|------|----------------|--------------|
| **規則一致性** | 規則明確，可預測 | 規則不明，隨機死亡 |
| **給予工具** | 給你足夠的工具應對挑戰 | 不給你工具卻要求你克服 |
| **預告** | 事先警告危險 | 突然襲擊，無預警 |
| **迭代時間** | 死後快速重來 | 死後損失大量進度 |
| **可用性** | 可以暫停、存檔、調整 | 不能暫停，自動存檔陷阱 |
| **難度曲線** | 平緩上升 | 突然陡峭（Difficulty Spike） |

📺 **The True Genius of Dark Souls II** 展示了另一種方式：**透過遊戲內機制而非選單選項來調整難度**。
- 遠程攻擊（讓你可以打帶跑）
- 召喚幫手（降低 Boss 難度）
- 生命保護戒指（死亡不掉魂）
- 玩家自己選擇用不用這些工具

📺 **Video Game Difficulty is Hard** 引入了「自適應難度」的概念：遊戲根據玩家表現自動微調難度。但更務實的做法是**提供模組化的輔助選項**（如 Celeste 的 Assist Mode），讓玩家自己決定開啟哪些輔助功能。

📺 **What's The Point Of Hard Games, Anyway** 分析了選擇提高難度的三個正當理由：
1. **增加成就感：** 困難讓勝利更有價值
2. **配合敘事主題：** 困難的戰鬥反映故事中的困境
3. **鼓勵特定玩法：** 高難度迫使玩家學習和運用系統深度

**如果以上三個理由都不成立，不要為了難而難。**

### C. 教學與引導

📺 **Why Tunic Hides Its Tutorial** 的核心教訓：

> 資訊遮蔽可以創造發現的樂趣，但必須讓玩家有足夠的基礎知識可以開始玩。

**好的教學設計原則：**
1. **讓玩家透過「做」來學習**，而不是透過「讀」
2. **展示，不要告訴** —— 用關卡設計教玩家怎麼玩
3. **給玩家安全的實驗空間**（如 Minecraft 的第一天沒有怪物）
4. **承認有些東西就是需要直接告訴玩家**（如 *Half-Life 2* 的 Alyx 會親自教你操作）

📺 **Easy Games - What Happened to Hardcore Games** 的洞見：玩家追求的其實不是「易上手」，而是**「深度」**（Depth）。好的教學讓玩家在其他玩家還在摸索時，你已經在探索深度了。

📺 **Negative Possibility Space** 的警告：**當你創造了玩家期望但沒有給予回報時，玩家會感到失望。**
- 如果你展示了一個鎖住的門，你必須讓它最終可以打開
- 如果你展示了一座山，你必須讓玩家可以爬上去或獲得解釋為什麼不能爬
- **承諾少一點，兌現多一點**

📺 **Progression Systems** 進一步指出：好的成長系統應該**引導學習曲線**——先給簡單的能力（增加玩家選項），再給需要組合的進階能力（增加策略深度）。

### D. 資訊揭露的平衡

📺 **Balatro's 'Cursed' Design Problem** 是一個警示故事：

> 如果玩家可以繞過你的設計選擇（即使需要耗費大量精力），你的設計就有漏洞。

**資訊揭露的光譜：**

| 完全隱藏 | 隱藏但可推算 | 提示但不完全 | 完全揭露 |
|---------|------------|------------|---------|
| 玩家無法得知 | 玩家可以自己算（如 Balatro 分數） | 間接線索 | 直接顯示 |
| 創造最大懸念 | ✗ 玩家的繞過行為破壞體驗 | 保持發現樂趣 | 失去懸念但獲得清晰度 |

📺 **What Makes a Game Feel Mysterious** 提供了四種資訊遮蔽技巧：
1. **鎖住的門：** 玩家知道目標在哪裡，只是需要找到鑰匙
2. **隱藏的規則：** 讓玩家透過實驗發現系統運作方式（如 *Baba is You*）
3. **未知的地圖：** 地圖需要玩家逐步探索才會顯示
4. **敘事謎團：** 故事情節需要玩家拼湊

📺 **Hidden Game Mechanics Design for the Human Psyche** 列出了遊戲設計中常見的「善意謊言」：
- **Coyote Time：** 離開平台邊緣後短時間內仍然可以跳躍
- **子彈補償（Bullet Magnetism）：** 子彈略為偏向目標
- **血量保護：** 最後一滴血時有短暫無敵
- **這些「作弊」讓遊戲感覺更好**，玩家即使知道也不會抱怨

📺 **The Power of Video Game HUDs** 提供了資訊設計的視角：
- **儀表功能（Dashboard）：** 顯示當前狀態（HP、彈藥、位置）
- **預覽功能（Preview）：** 顯示「如果...會怎樣」（如 Balatro 的分數預覽）
- **Diegetic UI（存在於遊戲世界內的 HUD）：** 如 *Dead Space* 的生命條在角色背上，*Metro Exodus* 的子彈在手上——提升沉浸感

**決定資訊等級的三個問題：**
1. 這個資訊會讓玩家做出「更好的決策」嗎？→ 如果是，就顯示
2. 這個資訊會「破壞懸念或驚喜」嗎？→ 如果是，就隱藏
3. 玩家有辦法繞過隱藏來獲得這個資訊嗎？→ 如果是，考慮重新設計

### E. 遊戲經濟設計

📺 **How Video Game Economies are Designed** 和 📺 **MMO Economies** 提供了經濟設計的基礎：

所有遊戲都有經濟系統，即使不是 MMO。五個基本元素：

| 元素 | 定義 | 例子 |
|------|------|------|
| **水龍頭（Faucet）** | 資源的來源 | 殺怪掉錢、任務獎勵、時間生成 |
| **庫存（Inventory）** | 資源的儲存 | 錢包、背包、銀行 |
| **轉換器（Converter）** | 資源轉換 | 商店（金錢→裝備）、合成系統 |
| **排水管（Drain）** | 資源的消耗 | 修裝費、稅收、消耗品、死亡扣款 |
| **交易者（Trader）** | 玩家之間的資源流動 | 拍賣場、交易系統 |

**核心平衡：** 如果水龍頭 > 排水管，通貨膨脹 → 資源失去價值。如果排水管 > 水龍頭，玩家感到資源匱乏的挫折。

📺 **This Psychological Trick Makes Rewards Backfire** 也警告 Overjustification Effect：當你給玩家外部獎勵去做他們本來就覺得有趣的事情時，會降低他們的內在動機。不要過度獎勵！

📺 **The Skinner Box** 的解決方案：用好奇心、精通、挑戰和故事來取代隨機獎勵。

## 第六步：原型驗證

> **目標：** 以最低成本驗證你的核心假設，而不是閉門造車三個月才發現不好玩。

### A. 做一個「可玩的原型」（1~7 天）

📺 **Prototyping, Vertical Slices, and Planning a Complex Game** 強調：原型不需要美術、不需要音樂、不需要故事。只需要：
- **核心機制可以操作**
- **能完整跑完一輪 gameplay loop**
- 花費時間：1~7 天

📺 **Most Steam Demos Suck** 的關鍵區分：

| | 原型（Prototype） | 試玩版（Demo） |
|--|----------------|--------------|
| 目的 | 驗證核心玩法 | 銷售遊戲 |
| 品質 | 方塊、白盒、placeholder | 打磨過、有美術有 UI |
| 對象 | 你自己和測試者 | Steam 玩家 |
| 時間 | 1~7 天 | 數週到數月 |

**不要混淆原型和試玩版。** 原型用醜陋的灰色方塊跑就可以了，試玩版才需要漂亮。

📺 **Level Design Approaches for Solo Devs** 的 Grayboxing 方法：先用簡單的幾何形狀搭建關卡的功能原型，測試跳躍距離、視野、路線後，再來做美化。

### B. 找真實玩家測試

📺 **Valve's Secret Weapon** 揭示了 Valve 的測試方法論：

1. **找 3~5 個不是你團隊的人**（最好是目標玩家群）
2. **不要教他們怎麼玩**，觀察他們如何摸索
3. **不給任何提示**，即使他們卡住
4. **觀察他們的表情和語言**：
   - 「咦？」→ 困惑（UI／引導有問題）
   - 「哇！」→ 驚喜（保留）
   - 「嗯...」→ 投入思考（好現象）
   - 打哈欠→ 無聊（核心問題）
5. **結束後問三個問題：**
   - 「你最喜歡的部分是什麼？」
   - 「你最困惑的部分是什麼？」
   - 「你還會想繼續玩嗎？」

📺 **Playing Like a Designer** 的偏誤認知提醒：作為設計師，你已經太了解自己的遊戲，無法客觀判斷。**你不會知道一個新玩家的體驗是什麼。** 這就是為什麼測試如此重要。

📺 **How to Make Anything Fun** 的 Flow 理論可以幫你診斷玩家的心流狀態：

| 玩家狀態 | 可能原因 | 解決方案 |
|---------|---------|---------|
| 焦慮（挑戰太高） | 難度曲線太陡 | 調整難度、給更多工具 |
| 無聊（挑戰太低） | 變化不夠、沒有新挑戰 | 引入新元素 |
| 心流（挑戰適中） | ✅ 找到了正確的平衡 | 確認並保持 |

### C. 迭代迭代迭代

📺 **Fail Faster** 的理念：**速度 > 完美**。

- 做一個很醜但可玩的原型 → 1 天
- 找 3 個人測試 → 1 天
- 根據回饋修改 → 2 天
- 再測試 → 1 天

一個 5 天的迭代循環比一個月閉門造車更有效。

📺 **How do Game Designers make their games FUN** 的迭代法：
1. 列出所有你認為「有趣」的部分
2. 讓測試者確認這些部分真的有趣
3. **刪除測試者認為無聊的部分**（即使你花了最多時間做它）

📺 **Kill Your Darlings** 的辯論法：當你需要決定保留還是刪除一個功能時，把你的論點當作「辯論」而非「爭論」：
- 爭論：我想要保留這個功能 → 情感驅動
- 辯論：這個功能是否服務於我們的設計目標？→ 理性驅動

### D. 注意「隱藏的玩家行為」

📺 **Balatro's 'Cursed' Design Problem** 和 📺 **How Game Designers Protect Players From Themselves** 共同揭示了一個現象：**玩家會以設計師預期之外的方式玩遊戲。**

- Balatro 的玩家用計算機／表格計算分數（不是設計師希望的「交叉手指期待」體驗）
- Binding of Isaac 的玩家用瀏覽器查物品功能（不是設計師希望的「神秘探索」體驗）
- 你的遊戲中，玩家會怎麼「作弊」來破壞你設計的體驗？

**解決方法：** 在測試時特別觀察玩家是否在**做一些你沒預期到的事情**。如果有，要嘛封堵這個漏洞，要嘛正式支援它。

📺 **The J.C. Penney's Effect** 也提醒：玩家的**感受**比事實更重要。即使你的數學完全平衡，玩家也可能因為感知偏差而覺得不公平。測試時要捕捉這些感受上的問題。

## 第七步：規模化與打磨

> **目標：** 在保持核心樂趣的前提下，擴展內容並打磨品質。

### A. 決定遊戲規模

📺 **Most Steam Demos Suck** 和 📺 **Prototyping, Vertical Slices** 提供了兩種風格：

| 方式 | 說明 | 適合 | 風險 |
|------|------|------|------|
| **Horizontal Slice** | 做很多內容但深度淺 | 內容驅動的遊戲（解謎、關卡） | 內容空洞重複 |
| **Vertical Slice** | 做少量內容但精心打磨 | 機制驅動的遊戲（Roguelike、策略） | 內容太少玩家很快膩 |
| **混合策略** | 先 Vertical 確認核心有趣，再 Horizontal 擴展內容 | 大部分商業遊戲 | 需要多次迭代 |

📺 **The 100 Games That Taught Me Game Design** 的精神：每款遊戲只需要一個核心設計概念。一個好的 Vertical Slice 只需要展示這個概念。

### B. 建立重複與變化的節奏

📺 **How (and Why) to Design for Repetition With Variety** 以音樂式結構理解遊戲節奏：

```
主題（核心循環） → 變奏（引入新元素） → 回歸主題（玩家再次掌握） → 新的變奏
```

**具體執行建議：**
- 每 2~3 分鐘引入一個新元素（地型、敵人、能力）
- 每 10~15 分鐘有一個小高潮（Boss、新區域、敘事轉折）
- 每 30~60 分鐘有一個重大變化（新機制、新世界）

📺 **The Disappointing Final Level** 警告：**結局通常令人失望，因為它是開發末期才處理的、缺乏測試的部分。**
- 在開發中期就建立結局原型
- 越早測試結局越好
- 確保結局的敘事弧線和機制弧線是一致的

📺 **The Skinner Box** 的替代方案：用**好奇心**驅動玩家繼續玩，而不是用隨機獎勵：
- 下一個關卡會有什麼新東西？
- 如果我組合這兩種能力會怎樣？
- 這個角色的秘密是什麼？

### C. 設計有意義的成長系統

📺 **Progression Systems - How Good Games Avoid Skinner Boxes** 提供了四個原則：

| 原則 | 解釋 | 壞例子 | 好例子 |
|------|------|--------|--------|
| **增加策略深度** | 新能力讓玩家有更多選擇，而不是更大的數字 | +10 攻擊力 | 新武器類型有不同攻擊模式 |
| **引導學習曲線** | 先給簡單的能力，後給需要組合的 | 一次給全部能力 | 逐步解鎖，每個都要熟練 |
| **獎勵有趣玩法** | 獎勵冒險和探索，而非重複作業 | 殺 100 隻怪解鎖 | 首次擊敗 Boss 解鎖 |
| **強化敘事** | 成長本身說故事 | 等級提升只是數字 | 每個能力有劇情背景 |

📺 **The 100 Games That Taught Me Game Design** 的精神：每個成長系統只需要聚焦一個設計概念。太多系統會彼此稀釋。

### D. 平衡（如果需要）

📺 **How Games Get Balanced** 提供了系統化的平衡方法：

1. **Trade-offs（取捨）：** 讓玩家在兩個或多個選項之間做取捨
   - ✅ 裝備 A 攻擊高防禦低 vs 裝備 B 攻擊低防禦高
   - ❌ 裝備 A 全面優於裝備 B（Nerf 或 Buff）

2. **Counters（克制）：** 剪刀石頭布式的循環克制關係
   - 戰士 > 弓箭手 > 法師 > 戰士
   - 關鍵是：**每個選擇都有明顯的弱點和優勢**

3. **數據驅動：** 統計勝率、選取率、使用率
   - 某角色勝率 > 60% → 太強（Nerf）
   - 某角色選取率 < 2% → 太弱（Buff）或太無聊（重新設計）

📺 **Perfect Imbalance** 提出了一個反直覺觀點：

> 不要追求「完美平衡」。刻意讓元素有細微的強弱差異，透過循環克制關係形成動態 Meta。

- **穩定均衡（Stable Equilibrium）：** 所有選項一樣強 → Metagame 一成不變 → 玩家無聊
- **週期性不平衡（Cyclical Imbalance）：** 環境變化 → 玩家適應 → 新策略出現 → 環境再變化

📺 **Power Creep** 的教訓：**不要用數值增加來解決問題。**

```
新玩家：「這個新武器攻擊力比舊的高 10 點！」
老玩家：（一年後）「舊武器完全沒用，新武器攻擊力從 100 變 200 了...」
```

解決方案：使用 **Incomparables（不可比較項）**——獨特的效果無法直接比較強弱。
- 不該做：攻擊力 +5 的新武器
- 該做：攻擊時有機率冰凍敵人的新武器（無法直接和 +5 攻擊力比較）

📺 **Strategic Uncertainty** 探討如何在遊戲後期保持不確定性：
- **戰術不確定性（Tactical Uncertainty）：** 我下一步該怎麼走？
- **策略不確定性（Strategic Uncertainty）：** 誰會贏？

好的遊戲在後期需要**策略不確定性**——即使你領先，也可能隨時翻盤。不要讓玩家在遊戲中期就知道自己穩贏了。

📺 **Balancing for Skill** 的 F.O.O.S. 策略：給新手一個簡單強力的初始策略，讓他們能快速上手。然後在後期引入更複雜但更強的路線。

### E. 退出點的設計

📺 **Exit Points** 提醒一個容易被忽略的設計面向：**玩家需要知道何時可以停下來。**

- **非自願退出點：** 遊戲當機、進度遺失、被強制踢出
- **自然退出點：** 關卡結束、存檔點、每日任務完成、得到休息提示

設計你的遊戲在每 20~30 分鐘有一個自然退出點，讓玩家可以安心放下遊戲。

## 第八步：行銷與發布

> **目標：** 讓你的遊戲被正確的玩家看到。

### A. 打造 Steam 頁面

📺 **How To Write Marketing Copy For Your Game** 提供了具體的文案結構：

**錯誤範例：**
> 「這是一款開放世界像素 RPG 遊戲，有 50 種技能、100 種武器和 200 種怪物。」

**正確範例：**
> 「你是一名在荒廢世界中求生的煉金術師。探索廢墟、收集材料、調配藥劑——但要小心，每一瓶藥都可能是解藥，也可能是毒藥。」

**核心原則：賣「體驗」不是賣「功能」。**

Steam 頁面的四個要素：
1. **標題：** 一句話讓人知道這是什麼遊戲（不是什麼創新概念）
2. **截圖：** 展示核心玩法，不是展示美術（玩家要看「我在做什麼」）
3. **預告片：** 15 秒內抓住注意力，展示 gameplay 為主
4. **介紹文案：** 描述玩家將**體驗**到什麼，而非列出功能清單

📺 **What Sells on Steam You Don't Need a Hook** 的結論：你的遊戲不需要一個「獨一無二」的鉤子。它只需要讓玩家感覺「我想體驗那個世界／氛圍／玩法」。

### B. 決定是否做試玩版

📺 **Most Steam Demos Suck** 的區分重訪：

| 做試玩版 | 不做試玩版 |
|---------|----------|
| 收集遊玩數據和回饋 | 保留完整遊戲的驚喜感 |
| 建立 Wishlist | 避免「試玩版品質太差」的逆行銷 |
| 需要 Vertical Slice 品質 | 需要更強的行銷策略（預告片、Demo reel） |

**如果你要做試玩版，記得：試玩版不是原型。** 它需要打磨過、有 UI、有教學。一個醜陋的試玩版比沒有試玩版更糟。

### C. 應對直播與社群

📺 **Twitch Kills Game Sales.. but not equally!** 的雙面數據：
- ~40% 因觀看直播而購買 → 你的遊戲需要「直播友好」
- ~39% 因觀看直播而放棄購買 → 你的遊戲可能看起來不好玩或有「雲通關」效應

**什麼樣的遊戲適合直播？**
- 有視覺上的高潮點（特效、Boss、驚喜）
- 有可討論的決策（策略分歧、道德選擇）
- 有隨機性（每次觀看體驗不同）
- 有「反應」的價值（恐怖遊戲、搞笑物理、驚喜時刻）

### D. 最終檢查：你的遊戲準備好了嗎？

📺 **The Disappointing Final Level** 的結局設計提醒：不要等到最後才處理結局。在開發中期就建立結局的 prototype，確保它在開發末期有足夠時間打磨。

📺 **Valve's Secret Weapon** 的最終提醒：在發布前再做最後一輪遊玩測試。確保沒有致命的 Bug，確保核心體驗沒有被打磨過程稀釋。

## 附錄：快速評估檢查表

### 核心檢查

- [ ] 我能在**設定時間**內完成嗎？（如不能，回到第一步）
- [ ] 我的**強項**能在這個遊戲中發揮嗎？（如不能，考慮換點子）
- [ ] 這個點子讓我**有熱情**做半年以上？（如沒有，不要做）
- [ ] 和市場上既有作品有**足夠的、有意義的差異**？（如沒有，回到第二步）

### 玩法檢查

- [ ] 核心玩法循環**只用一句話**就能說清楚？
- [ ] 這個循環**關掉聲音、拿掉美術、去掉獎勵**仍然有趣？
- [ ] 玩家能在 **5 分鐘內**理解基本操作？
- [ ] 有至少 **2~3 種樂趣類型**（MDA）？

### 規模檢查

- [ ] 如果只能做 1 個月，我能做出**可玩的 MVP**？
- [ ] 我已經確定哪些功能是**類型標準**、哪些是**創新**？
- [ ] 我準備好**砍掉**不必要功能？（砍什麼？寫下來）

### 玩家檢查

- [ ] 我知道**「誰不會」**玩這個遊戲？
- [ ] 我找到了 **3~5 個目標玩家**作為測試者？
- [ ] 我的難度設計是**挑戰而非懲罰**？

### 市場檢查

- [ ] 我的 Steam 頁面描述的是**體驗**而不是功能？
- [ ] 我的遊戲有至少一個「**直播時刻**」——能讓觀眾驚呼或討論的點？
- [ ] 我知道發布後**第一個月的行銷計畫**是什麼？

## 設計原則總結

從 98 篇影片中提煉出的 10 條核心原則：

1. **限制是自由的來源**（Design by Constraint）
2. **先找有趣的互動，再圍繞它設計一切**（Nintendo）
3. **玩家購買體驗，不是功能**（MDA Framework）
4. **越快失敗，越早知道怎麼修正**（Fail Faster）
5. **測試者的回饋比你的直覺更重要**（Valve）
6. **不要優化到沒有樂趣**（Perfect Imbalance）
7. **挑戰好、懲罰不好**（When Difficult Is Fun）
8. **教學應該藏起來，但不能不存在**（Tunic）
9. **空白和節奏和內容一樣重要**（Ma）
10. **你知道你的遊戲不該做什麼，比該做什麼更重要**（Subtractive Design）
