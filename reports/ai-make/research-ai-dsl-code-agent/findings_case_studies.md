# Research Findings: AI Code Agents Building Domain-Specific Languages (DSLs)

**Date:** 2026-05-01
**Scope:** Real-world case studies of LLMs creating DSLs from requirements

---

## 1. Basalt — AI-First Backend DSL

- **Project:** https://github.com/DiegoDev2/Basalt
- **Domain:** Backend API code generation
- **Context:** Basalt defines a declarative DSL (`.bs` files) for describing backend services — database schemas, API endpoints, auth, roles. An AI agent (Claude, ChatGPT, Gemini) reads the Basalt spec and produces a `.bs` file from natural language requirements. Basalt's CLI then compiles that into production-ready TypeScript (Express/Fastify/Hono + Supabase).
- **What worked well:**
  - Token efficiency: ~10 lines of DSL vs ~200 lines of TypeScript (up to 20x cheaper API calls).
  - Consistency: every generated project follows the same structure and conventions.
  - Reliability: the closed/constrained DSL means the AI cannot hallucinate imports or invent architectures — "if it's valid `.bs`, the output is valid code."
  - Explicit AI generation guidelines in the spec (e.g., "Output only the `.bs` file content, no explanations, no markdown fences.").
- **What didn't:**
  - Early-stage project (v0.1.0) — only one template (Express + TypeScript + Supabase).
  - The `basalt ai` command (natural language → `.bs`) was still on the roadmap, not yet built.
  - Relies on the AI having access to the spec document; no built-in guardrails if the model deviates.

## 2. Rubric — DSL for Constraining AI-Generated Code

- **Project:** https://github.com/graciolli-f/Rubric (11 stars)
- **Domain:** Architecture enforcement in JavaScript/TypeScript projects
- **Context:** Rubric is a constraint-driven DSL for enforcing clean architecture. Developers seed a `rubric/` folder with `.rux` specification files and a `.cursorrules` file. When a user prompts an AI agent (Cursor IDE) with a feature request, the agent automatically: generates `.rux` constraint files, validates them, plans code from the constraints, generates the code, validates the output, and iterates on failures.
- **What worked well:**
  - Automated validation loop: the agent self-corrects by running `node rubric/validate.js` and fixing errors.
  - Bug checklist: after generation, the agent runs through `rubric/RUBRIC.md` and creates `rubric/BUGS.md`.
  - Practical example of an AI agent collaboratively evolving a DSL with a human.
- **What didn't:**
  - Token overhead: "Using Rubric in Cursor increases number of tokens used in a single request."
  - Currently Cursor-only (relies on `.cursorrules` integration).
  - The DSL is tightly coupled to the AI agent's workflow, not a standalone language.

## 3. SWAN-DSL — AI-Generated UI Interaction DSL

- **Project:** https://github.com/NAIV-Dev/swan-dsl (AI-generated)
- **Domain:** UI interaction modeling (declarative, framework-agnostic)
- **Context:** SWAN-DSL is a full specification for describing user interfaces as "hierarchical interaction graphs with explicit navigation states." The entire DSL spec and implementation were generated by AI, documenting pages, components, navigation flows, tables, charts, queries, and validation phases. It includes a complete EBNF grammar and 14 static semantic rules.
- **What worked well:**
  - Comprehensive spec: the AI produced a well-structured DSL with types, EBNF grammar, semantic rules, validation phases, and multiple compilation targets (HTML, React, Flutter, SwiftUI).
  - Framework-agnostic design: "Pages are states. Components are behavior. Flows are explicit. Navigation is centralized."
  - The DSL guarantees static analyzability, explicit navigation graphs, and AI-safe generation.
- **What didn't:**
  - No evidence of real-world usage — appears to be a speculative/educational project.
  - No parser or compiler implementation beyond the spec document itself.
  - "This code is AI-generated" in the repo description, suggesting it was a proof-of-concept.

## 4. MCP-DSL — Declarative Language for Model Context Protocol

- **Project:** https://github.com/jmajerus/MCP-DSL
- **Domain:** Model-Context Interaction (AI agent orchestration)
- **Context:** MCP-DSL is a DSL for expressing model-context interaction logic in AI systems. The author collaborated with ChatGPT to draft formal grammars (PEG.js → ANTLR), explore language design patterns, map MCP features to DSL constructs, and build code generators targeting Python, Rust, Go, TypeScript, and Kotlin.
- **What worked well:**
  - Rapid prototyping: "With the help of ChatGPT, I rapidly sketched out an ambitious vision."
  - Multi-language code generation from a single declarative spec.
  - Built-in support for modern MCP concepts: tools, resources, prompts, agents, streaming, auth.
- **What didn't:**
  - **Project paused.** The author realized the MCP specification had evolved beyond their early assumptions — "I hadn't realized that the MCP specification had already evolved far beyond its initial form."
  - Classic DSL anti-pattern: designing the DSL before the underlying protocol was stable.
  - The project is an interesting case study of the limits of AI-assisted DSL design: the AI helped draft a beautiful language, but couldn't warn the author that the foundation was shifting.

## 5. AIM (Assembler In a Monad) — AI-Adjacent Haskell DSL

- **Project:** https://github.com/raaz-crypto/aim
- **Domain:** Assembly code generation
- **Context:** AIM is a Haskell DSL for generating assembly language, inspired by Daniel Bernstein's qhasm. It uses Haskell's monadic structure to embed assembly generation as a DSL, leveraging Haskell functions as macros. Marked as "AI-generated" at a high level (found via `ai generated dsl` search).
- **What worked well:**
  - Embedded DSL approach gives full host-language power (macros = Haskell functions).
- **What didn't:**
  - Abandoned since 2014 — predates modern LLMs entirely.
  - Only tangentially related to AI agents: it's a DSL that could be used *by* AI, not one *created* by AI.

## 6. Aiken Codegen — Haskell DSL for Cardano Smart Contracts

- **Project:** https://github.com/paolino/aiken-codegen
- **Domain:** Blockchain (Cardano) — generating Aiken smart contract source code
- **Context:** A Haskell DSL that generates Aiken (a Cardano smart contract language) source code. Found via the `ai generated dsl` GitHub search.
- **What worked well:**
  - Type-safe code generation using Haskell's type system guarantees valid Aiken output.
- **What didn't:**
  - Limited adoption (0 stars). Traditional embedded DSL approach, not LLM-driven.

## 7. LLM-DSL-Runtime-Emacs — DSL Runtime for LLM-Generated Code

- **Project:** https://github.com/kroq86/llm-dsl-runtime-emacs
- **Domain:** Safe execution of LLM-generated code in Emacs
- **Context:** An Emacs runtime for executing LLM-generated declarative DSL safely on real code via validated query/edit plans. The DSL acts as an intermediate representation between the LLM's intent and actual file modifications.
- **What worked well:**
  - Safety layer: validates LLM output before executing on real code.
  - DSL as an IR (intermediate representation) pattern — a promising architecture for AI code agents.
- **What didn't:**
  - Very niche (Emacs ecosystem). Early-stage project.

## 8. ai-dify-dsl — Typed Parser for Dify Workflow DSL

- **Project:** https://github.com/kevinten-ai/ai-dify-dsl
- **Domain:** AI workflow orchestration (Dify platform)
- **Context:** A typed parser and generator for Dify workflow DSL — parse, edit, optimize workflows locally with Python, then generate back to Dify-importable format. Positioned as an AI-assisted workflow editing tool.
- **What worked well:**
  - Round-trip engineering: parse → edit → regenerate back to the target platform.
  - Python-based, accessible to AI agents and data scientists.
- **What didn't:**
  - Tightly coupled to Dify's specific workflow format. Not a general-purpose approach.

## 9. AI Excalidraw — Natural Language to Hand-Drawn Diagrams

- **Project:** https://github.com/ramacuy/ai-excalidraw
- **Domain:** Diagram generation from natural language
- **Context:** Generates hand-drawn style diagrams using AI from natural language descriptions. The system acts as a "DSL generator" where the output format (Excalidraw's schema) is essentially a visual DSL produced by an LLM.
- **What worked well:**
  - Real-time rendering with multi-session support.
  - Agent-based TypeScript implementation.
- **What didn't:**
  - Not a traditional programming DSL — more of a visual output format.

## Key Patterns & Takeaways

### What Works Well
1. **DSL as an intermediate representation (IR):** The most successful pattern is using a constrained DSL as a bridge between natural language and generated code. Basalt, Rubric, and LLM-DSL-Runtime-Emacs all follow this pattern.
2. **Validation loops:** Rubric's self-validating workflow (generate DSL → validate → fix → generate code → validate → iterate) is a strong design pattern for AI + DSL workflows.
3. **Token efficiency:** Basalt reports up to 20x cost reduction by having the AI write DSL instead of full code.
4. **Closed/constrained grammars:** When the AI can only produce valid DSL tokens, the output is inherently reliable (Basalt's design principle).

### What Doesn't Work
1. **Building DSLs on unstable foundations:** MCP-DSL was paused because the underlying MCP spec evolved faster than the DSL could adapt.
2. **Token overhead:** Rubric acknowledges that the DSL workflow increases token usage — a real cost for users.
3. **Speculative DSLs without implementations:** SWAN-DSL and several other projects produced beautiful specs but no working compilers.
4. **IDE lock-in:** Rubric works only in Cursor; several other tools are tied to specific editors or platforms.
5. **Abandonment:** Many projects (AIM, SWAN-DSL, MCP-DSL) are paused or abandoned — this space moves very fast.

### Recommended Patterns for AI + DSL Projects
- Design the DSL to be both human-readable and LLM-generatable from a single spec document.
- Include explicit "AI Generation Guidelines" in the spec (as Basalt does).
- Build a validation loop: generate DSL → validate → fix → generate output → validate.
- Keep the DSL small and stable before expanding — the biggest failures came from over-ambitious scope.
- Consider the DSL-as-IR pattern for separating AI intent from implementation.
